# Prototype Build Plan — True Classic Paid Media Intelligence

A practical, buildable-in-an-afternoon plan for the 15-minute live demo. This narrows the
full production architecture (see `01_architecture.md`) down to a single Streamlit app that
runs end-to-end: **ingest → unify → reconcile → forecast → optimize → approve → explain**.

> **Rule for this build:** a narrow, *real, runnable* slice beats a broad mockup. Everything
> below is designed to run live on a laptop in under a minute, with the optimization logic
> genuinely live and only platform writes stubbed.

---

## 1. Scope & guiding decisions

| Decision | Choice | Why |
|---|---|---|
| Platforms | Meta + Google only | Brief's minimum bar; enough to show cross-platform unification |
| UI | Streamlit (single app, multi-tab) | Fastest path to a runnable, narratable UI in Python |
| Forecast model | `XGBoost`/`LightGBM` (or `scikit-learn` GBR fallback) + simple seasonality feature | Tabular, nonlinear, defensible; fast to train on synthetic data |
| Saturation curve | Hill / log response curve fit per channel | Gives the optimizer marginal ROAS, not just a point forecast |
| Optimizer | Marginal-ROAS "water-filling" allocator under constraints | Deterministic, auditable, explainable — not an LLM |
| LLM | One call to narrate the recommendation (OpenAI/Anthropic), with rule-based fallback | Real LLM value (explanation) without hallucination risk on numbers |
| Data | Synthetic CSVs generated by a seed script | Transparent about what's real vs mocked |

**What's REAL logic:** schema normalization, SKU reconciliation, feature building, model
training + forecast, saturation-curve fit, the constrained allocator, diminishing-returns
detection, KPI math (Blended ROAS / NC-CPA / utilization).

**What's MOCKED/STUBBED:** the source CSVs (synthetic, seeded), the LLM call has a
deterministic fallback, and the "execute / push to platform" button only writes to a local
audit log — no real Meta/Google API write.

## 2. Exact files & folders to create

```
TrueClassic/
├── README.md                        # what's real, what's mocked, what's next
├── requirements.txt                 # streamlit, pandas, numpy, scikit-learn, xgboost, scipy, plotly, (openai|anthropic)
├── .env.example                     # LLM_API_KEY=...  (optional; app runs without it)
├── docs/
│   ├── 01_architecture.md           # production architecture (already discussed)
│   └── 02_prototype_build_plan.md   # this file
├── data/
│   ├── generate_mock_data.py        # seeded synthetic CSV generator (run once)
│   ├── meta_ads_export.csv          # generated
│   ├── google_ads_export.csv        # generated
│   └── sku_catalog.csv              # source-of-truth SKU list + crosswalk (with 1 gap)
└── app/
    ├── streamlit_app.py             # entrypoint: tab layout + state, wires modules together
    ├── ingestion.py                 # load raw Meta + Google CSVs
    ├── normalize.py                 # map each schema -> unified canonical dataframe
    ├── reconcile.py                 # SKU identity resolution + flag the mismatch
    ├── forecast.py                  # features, model train/predict, confidence interval
    ├── curves.py                    # fit spend->response saturation curve per channel
    ├── optimize.py                  # marginal-ROAS allocator + diminishing-returns detector
    ├── kpis.py                      # Blended ROAS, NC-CPA, budget utilization
    ├── explain.py                   # LLM narration of a recommendation (+ rule-based fallback)
    └── audit_log.py                 # append approved/rejected decisions to data/audit_log.csv
```

Each `app/*.py` module is independently testable and exposes a small, typed function surface
so the Streamlit layer is thin glue.

## 3. Data columns needed

### `meta_ads_export.csv` (Meta-style schema)
| Column | Notes |
|---|---|
| `date` | YYYY-MM-DD |
| `campaign_name` | encodes funnel stage, e.g. `TC_Prospecting_Tees`, `TC_Retargeting_Tees` |
| `adset_name` | |
| `product_id` | **Meta catalog ID** (e.g. `FB-1001`) — intentionally different key format |
| `spend` | USD |
| `impressions`, `clicks` | |
| `purchases` | conversions (7d-click/1d-view attributed) |
| `purchase_value` | platform-reported revenue |
| `new_customers` | for NC-CPA |

### `google_ads_export.csv` (Google Ads-style schema)
| Column | Notes |
|---|---|
| `day` | different date column name on purpose |
| `campaign` | e.g. `Search_Prospecting_Tees`, `PMax_Tees` |
| `ad_group` | |
| `item_id` | **Google merchant item_id** (e.g. `G-2001`) — different key format |
| `cost` | USD (note: "cost" not "spend") |
| `impr`, `clicks` | different impression column name |
| `conversions` | |
| `conv_value` | revenue |
| `new_cust` | |

### `sku_catalog.csv` (source of truth + crosswalk)
| Column | Notes |
|---|---|
| `sku` | canonical SKU, e.g. `TEE-CREW-BLK` |
| `product_name` | |
| `meta_product_id` | maps to Meta `FB-xxxx` |
| `google_item_id` | maps to Google `G-xxxx` |

> **The intentional mismatch:** leave ONE product's `google_item_id` blank (or wrong) in the
> catalog — e.g. a SKU that exists in the Google export but has no crosswalk entry. The
> reconciliation step must *detect and surface* this, not silently drop it.

### Canonical unified schema (output of `normalize.py` + `reconcile.py`)
```
date | platform | campaign | funnel_stage | sku | product_name |
spend | impressions | clicks | conversions | revenue | new_customers | match_status
```
`funnel_stage` ∈ {prospecting, retargeting} parsed from campaign name.
`match_status` ∈ {matched, unmatched_no_crosswalk, ...} drives the data-quality flag.

## 4. Streamlit screens (tabs) — what each shows

A single app with 6 tabs, walked left-to-right = the demo narrative. Tab 0 is the landing
"so what" view; Tabs 1–5 are the drill-down into how each number was produced.

### Tab 0 — Overview (Command Center)  *(executive / buyer landing)*
The at-a-glance dashboard a marketer opens first. Headline KPI cards across the top, the single
top recommendation in the middle, and KPI status vs target.

- **KPI cards row** (computed live from the unified data, not hardcoded):
  - **Today's Spend** — e.g. `$24,000`
  - **Today's Revenue** — e.g. `$103,000`
  - **Blended ROAS** — e.g. `4.29x` with a green/red chip vs the 4.0x target
  - **NC-CPA** — actual vs ceiling, color-coded
  - **Budget Utilization** — e.g. `92%` (flags caps-hit-by-noon underspend)
- **Top recommendation banner** — the single highest-impact move, surfaced from the optimizer:
  > **Move $1,800   Google → Meta   ·   Expected gain +$7,100**

  with a one-line LLM/rule-based reason and a **"Review in Allocation"** button that jumps to
  Tab 4.
- **Mini trend** — small sparkline of blended ROAS over the last N days.
- **Flags strip** — count of data-quality issues (e.g. "1 unreconciled SKU") and
  diminishing-returns alerts, each linking to the relevant tab.

> Every number here is derived from the same `kpis.py` + `optimize.py` outputs used in the
> deeper tabs — the Overview is a *view*, never a separate source of truth.

### Tab 1 — Ingestion & Unification  *(M1)*
- Two side-by-side previews of the **raw** Meta and Google CSVs (show the schemas don't match: `spend` vs `cost`, `date` vs `day`, different ID formats).
- Button: **"Unify"** → shows the single canonical dataframe.
- A **Data Quality panel**: row counts per platform, date-range coverage, and a red flag:
  *"1 SKU from Google could not be reconciled (no crosswalk entry) — excluded from spend rollups, surfaced for review."*

### Tab 2 — Forecast  *(M2)*
- SKU-set + channel selector.
- Line chart: historical revenue + forecast with a **confidence band**.
- Per-channel comparison (Meta vs Google) showing different spend-response behavior.
- Small metrics row: backtest **MAPE/accuracy** so the viewer knows how much to trust it.

### Tab 3 — Efficiency Curves  *(M2→M3 bridge)*
- Per-channel **saturation curve**: spend on x, revenue/ROAS on y, with the **current spend
  point** marked and the **marginal ROAS** annotated.
- Visually shows diminishing returns and where each channel saturates.

### Tab 4 — Allocation Recommendation  *(M3)*
- Inputs visible (the brief requires this): total budget, current per-channel spend, ROAS
  floor (default 4.0x), NC-CPA ceiling, max daily change %.
- Table: **current → recommended** spend per channel, with expected ΔROAS and the binding
  constraint per row.
- **Diminishing-returns callout:** e.g. "Meta retargeting marginal ROAS = 2.1x < 4.0x floor →
  recommend cut $X; reallocate to Google PMax (headroom to 5.3x)."
- KPI before/after panel: Blended ROAS, NC-CPA, utilization %.

### Tab 5 — Approval View (Marketer / Buyer)  *(M3 human-in-the-loop)*
- Each recommendation as a row with **Approve / Reject** toggles and editable target spend.
- **LLM-generated explanation** box per recommendation (plain-English rationale built from the
  actual numbers).
- Constraint sliders the marketer can adjust → **"Re-run"** recomputes the allocation live.
- **"Approve & Execute (stubbed)"** → writes the decision to `audit_log.csv` and shows a
  confirmation. No real API write.

## 5. Real logic vs mocked — explicit map

| Component | Status | Detail |
|---|---|---|
| Source CSVs | **Mocked** | Seeded synthetic generator; deterministic so demo is repeatable |
| Schema normalization | **Real** | Genuine column mapping + funnel-stage parsing |
| SKU reconciliation | **Real** | Crosswalk join + fuzzy fallback; real detection of the 1 mismatch |
| Attribution | **Simplified** | Use platform-reported revenue; note in README that prod reconciles to Shopify |
| Forecast model | **Real** | XGBoost/GBR trained live on the synthetic history |
| Confidence interval | **Real** | Quantile or residual-based band |
| Saturation curves | **Real** | Curve fit (`scipy.optimize`) per channel |
| Optimizer | **Real** | Constrained marginal-ROAS allocator |
| Diminishing-returns detection | **Real** | Threshold on marginal ROAS |
| KPI math | **Real** | Blended ROAS / NC-CPA / utilization computed from data |
| LLM explanation | **Real call, safe** | LLM narrates given numbers; rule-based fallback if no API key |
| Platform execution | **Stubbed** | Writes to local audit log only |

## 6. Build order (so it's runnable at each step)

1. `requirements.txt` + `generate_mock_data.py` → produce the 3 CSVs (include the mismatch).
2. `ingestion.py` + `normalize.py` → unified dataframe; verify in a scratch run.
3. `reconcile.py` → confirm the 1 mismatch is flagged, not dropped.
4. `streamlit_app.py` Tab 1 → first runnable UI milestone.
5. `forecast.py` + `curves.py` → Tabs 2 & 3.
6. `optimize.py` + `kpis.py` → Tab 4 (the core decision).
7. `explain.py` + `audit_log.py` → Tab 5 (approval + LLM).
8. **Tab 0 Overview** → built last, since it just composes `kpis.py` + `optimize.py` outputs
   into headline cards + the top recommendation banner. Make it the default landing tab.
9. `README.md` → real/mocked/next.

Each step leaves the app launchable (`streamlit run app/streamlit_app.py`), so a partial build
still demos.

## 7. The 15-minute demo script (what to say)

**0:00–1:30 — Frame it + Overview (Tab 0).** "True Classic runs paid media across Meta, Google,
Amazon, Microsoft — nothing's unified, attribution conflicts. This prototype does a real
end-to-end slice on Meta + Google. Here's the command center a buyer opens: today's spend
$24k, revenue $103k, blended ROAS 4.29x — green, above our 4.0x target. The system's top move
right now: shift $1,800 from Google to Meta for an expected +$7,100. Everything you'll see is
computed live; only the platform writes are stubbed. Let me show you how it gets to that
number." → click into Tab 1.

**1:30–4:00 — Tab 1, Ingestion.** "Here are two raw exports — notice the schemas don't match:
`spend` vs `cost`, `date` vs `day`, and the product IDs are in totally different formats. I hit
Unify and it normalizes into one canonical table. Critically — see this red flag — one Google
SKU has no crosswalk entry, so instead of silently dropping it, the system surfaces it for
review. That's the M1 reconciliation requirement."

**4:00–7:00 — Tabs 2 & 3, Forecast + Curves.** "This is an XGBoost model trained live on the
history, with a confidence band so you know how much to trust it — backtest MAPE is shown here.
Meta and Google behave differently, which you can see in their spend-response curves. The marked
point is current spend; this annotation is the *marginal* ROAS — the return on the next dollar.
That's what the optimizer actually uses."

**7:00–11:00 — Tab 4, Allocation.** "Now the decision. All inputs are visible — total budget,
current spend, the 4.0x ROAS floor, the NC-CPA ceiling, max daily change. The allocator moves
each marginal dollar to the highest-return channel until a constraint binds. Here it detects
Meta retargeting marginal ROAS has fallen below the 4.0x floor — diminishing returns — so it
recommends cutting it and reallocating to Google PMax, which still has headroom. Watch blended
ROAS go from X to Y."

**11:00–14:00 — Tab 5, Approval + LLM.** "A marketer never gets a black box. Each recommendation
has a plain-English explanation — that's an LLM narrating the *actual numbers* the optimizer
produced, never inventing them, with a rule-based fallback if the API is down. They can adjust
constraints with these sliders and re-run live, then approve or reject per line. Approve &
Execute is stubbed — it writes to an audit log instead of calling the Meta/Google APIs."

**14:00–15:00 — Close + AI workflow.** "Stack: Streamlit, pandas, XGBoost + scipy curve fit, a
deterministic constrained optimizer, and an LLM only for explanation. I built it with
Cursor/Claude — happy to show prompts and where I intervened. With two more weeks I'd add Amazon
+ Microsoft connectors, reconcile attribution against Shopify, and wire real platform writes
behind the approval gate."

## 8. Talking points held in reserve (for Q&A)

- **Why an explicit optimizer, not an LLM, for budget math?** Determinism, auditability, and
  hard constraint satisfaction — money decisions must be reproducible.
- **Hallucination control:** the LLM only templates/explains numbers passed to it; raw numbers
  are shown alongside; rule-based fallback exists.
- **Attribution:** prototype uses platform-reported revenue; production reconciles to Shopify
  as source of truth (where Northbeam/Triple Whale or an MMM would plug in).
- **Scaling to thousands of levers:** the same allocator generalizes from channels to
  campaign/ad-set grain; the canonical schema is the contract that makes that swap clean.
